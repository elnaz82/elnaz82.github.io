Quest eCommerce website Redesign

May 2018 |

Project Overview

The purpose of this project were to:

  • create a linear user journey
  • simplify checkout process and create a delightful experience while purchasing products
  • align style with Quest branding
  • test and track engagement on new design
  • iterate on design and processes based on quantitative research studies

My Role

This project has been done entirely by myself. Some of the main activities are:

  • Understand the current user flow and its flaws based on actual users feedback/existing data
  • Work closely with eCommerce colleagues and stakeholders to capture requirements for different business units
  • Research best practices using Baymard Institute resources
  • Create new user flow based on findings
  • Sketch ideas for the checkout process
  • Create pixel-perfect mockups using Sketch app & Quest design library design components
  • Create custom designs to address the special needs of certain products
  • Review and revise several times based on feedback
  • Specifications hand over to Quest's 3rd party vendor developers for implementation
  • Test and match design to make sure all specs are met and functionality works as defined

Issues

  • Confusing product information (pricing, license options, currency type, etc.)
  • Information overload during user journey
  • Slow checkout process
  • Old Dell style that didn't reflect new Quest branding

Solution

  • Conduct research to find industry best practices and apply to the current flow to improve the user experience
  • Understand user pain points and come up with solutions using existing data
  • Create an improved user flow and review before starting the project to hash out any confusions or issues
  • Focus on functionality first rather than aesthetic
  • Use our existing components from Quest design library to make the process quicker
  • Test new design and use benchmarking methods to measure success
  • Iterate and improve user experience based on findings

Challenges

  • Planning and gathering accurate requirements
  • Lack of budget to conduct research
  • Find a reliable research resource for eCommerce to benefit from findings
  • Custom solutions for certain products with more complex purchasing options which can increase the implementation cost
  • Communicate requirements to a 3rd party vendor for development
  • Ship first, test later :((

Design Process Overview

Planning & Gathering Requirements

The main goal of this project was to not only align our eCommerce website branding with Quest, but also find solutions for a number of pain points that were experienced during customer journey map while purchasing a product online. My ultimate goal was to talk to both stakeholders and customers and find a common area between the two in order to satisfy both parties.

After interviewing some of my eCommerce colleagues who directly communicate with our customers, and also the stakeholder, I was able to identify almost all of the problems. I'm going to explain them while showcasing new layout in "Research & Mockup" section of this page.

User Flow

Before starting to sketch any ideas, I created a new user flow based on my findings so that I could validate my assumptions and identify any steps missed or have issues sooner than later. That way I could save some time during design by making sure my solution would target solving the right problem.

My goal was to go through the whole process with stakeholders using this user flow and eliminate any uncertain assumptions later in the project.

Research & Muckup

Assuming that you are new to this product, I figured I could best explain by showing before & after look of main pages and then go through existing issues, the proposed solutions, and the rationale behind them.

I'm pretty sure you can guess which one is before and which one is after!

Catalog Listing Page

Research revealed that:

  • Most eCommerce sites put country first and shopping cart after in their header area.
  • Breadcrumb didn't add any value to the page in terms of navigation
  • Filters weren't so intuitive, considering that we have a long list of products (13329px tall when I grabbed a screen capture of the page!!!)
  • All the information for each product placed right on the listing page which overwhelmed users and made some confusion
  • Many users that come to the site from different regions weren't sure about the currency unit for pricing (is it USD? CAD? AUD?)
  • Licensing options were vague to the users and don't expose the actual offerings of each option
  • Very few people used the Assistance box at the bottom of the page

In this new layout I tried to solve some of the issues based on industry best practices by applying the following updates:

  • I switched country and shopping cart order, using our new Quest iconography
  • Removed the breadcrumb since they weren't adding any value anyways
  • Using our hero banner component from Quest design library, I dedicated a promotional area for popular products. This way I could meet one of the requests of the stakeholders.
  • Worked with copywriters to provide SEO friendly title and subtitle
  • Used the same filters we use on Quest listing page which are performing well and replaced with tabs. Also got rid of the alphabetical filter and ordered products alphabetically by default in the list. That way if we had only one product for a particular letter, the rest of the space wouldn't be wasted.
  • Made filter area sticky, so that they will be accessible no matter where users are on the page
  • Added back to top button for better navigation in this long page
  • Using card view concept for each product, I tried to provide a small amount of information to the users in this entry page and incorporated the rest of the information in the product details page
  • Added currency type abbreviation to pricing across the site to eliminate the confusion
  • Combined assistance and security logos with footer to clean up the bottom portion of the page
Product Details Page

Research revealed that:

  • Page title is redundant and doesn't mean anything
  • Every time a product tour is updated on Quest.com, there was a manual process going on to sync tour on eCommerce site
  • Products demo video link got 3% more clicks than product tour
  • Product name and it's short description are lost in the content
  • Information related to price have proximity issue
  • Almost all eCommerce sites give users the ability to specify quantity when they want to add an item to their shopping cart and view the pricing reflected accordingly. In this case users could select quantity after the fact.
  • I also noticed there are some discount opportunities for enterprise sales. However, there is no obvious way to get in touch with sales people or be aware of this opportunity on the page.

Here are some of the changes to clean up and make information easier to scan:

  • Got rid of the page main title
  • Used bigger and bolder text for product name and description, showing it at the center of the page
  • Looking at the old user flow I notice when people land on eCommerce product landing page they have already completed their research and awareness phase and they are likely to make a purchase, so I got rid of product tour to also avoid the manual process each and every time a product has an updated tour. However, I included a link back to the actual product page on Quest.com, just so if the users need to go back and check out more granular information about the product they can.
  • Replaced watch video link with an actual play in place video
  • Using the new pricing style, I decided to add a label at the bottom to place any additional information tied into it. This way I would solve the proximity issue.
  • I have also used more prominent CTA for "Add to Cart" using our primary orange Quest color since that's the most important thing I wanted our users to do!
  • Introduced "Contact Sales" CTA with the same style label I used for price, indicating the purpose of the button to bring some awareness about the purpose of the CTA.

Based on the articles I read on Baymard Institute's site I learned that interestingly some people remove the items from their cart using quantity drop down! Simply by clicking on "0" or "Remove" option. I incorporated that functionality into the drop-down and also allowed custom data input with proper validation to make it as flexible as possible for the users to specify a quantity and see the price accordingly reflected right on the product details page.

Product Details with Different Versions & License Options

One other issue that customers had was the confusion they faced while picking a version of a product and the licensing option they needed for it.

I think the best way to solve this is to break it down into two sub-tasks by creating a page addressing only different versions and then provide the license options based on the selection. To achieve this functionality I have created following sub-pages to improve the user flow at this stage of the funnel:

On this page I'm focusing on visualizing data tied to each of the versions of this product, using:

  • A chart from the product's datasheet which is normally embedded on the page as a PDF file, so that users can compare different functionalities and pick the right version
  • I have also tried to put an informative panel in case they are still not sure and need to talk to somebody.

This chart comparison pattern has been A/B tested on Quest.com and succeeded to win. It has been implemented on Toad for Oracle trial download page.

When a version is selected, the user is taken to the next step where they can view and select the licensing options available for that particular version of the product.

There are some benefits built into this new layout. The ultimate goal is to include every piece of information so that users can select the best option for their needs:

  • Every licensing option is clearly defined in this tiling approach, which is so popular if you visit the pricing page of many products based on my competitive research
  • If there is any promotion available for any of these options they will show so that they can save while making a purchase. A recommended badge has also been added for the option recommended by Quest.
  • There is a quantity drop-down and add to cart CTA for each option so that users can immediately add them to their shopping cart.
Shopping Cart & Checkout

Based on existing data, users are confused to take all the actions they need to complete their purchase in a single step where they have to not only review their cart but also complete the required information for payment.

As you can tell in this page we have shopping cart followed by some promotions tied to the products in the cart (which I included in the details page to grab their attention before adding to the cart). After that, we have checkout which is basically a single step to input all the remaining data.

I broke down shopping cart and checkout into two separate tasks taking a linear design approach so that users can focus on one at a time and the process doesn't seem so tedious and lengthy.

Based on my finding from Baymard Institute accordion steps style checkout which collapses into summaries, would be the most efficient way of doing a checkout process in an eCommerce site. So I took the same approach using my interaction design skills to provide proper feedback every time the user completes a step, along with showing the summary of the data entered and an ability to edit on the same page if they need to, without leaving the page.

I have also included a summary box which will provide their order summary in case they want to check any information related to pricing or products in their cart down the road.

As you can see in this screenshot:

  • I've added a navigation bar indicating the number of steps for checkout. My goal here is to give users a visual navigation of where they are in the process
  • If the task both is not active and hasn't been completed, it will show as light grey in disabled mode
  • Active step will be highlighted by Quest orange
  • When the step is completed successfully the number of step will be replaced with a check mark and the step becomes dark grey
  • Additionally, users can edit any step using the edit hyperlinks associated with each of them

Once all the steps are completed, users can review and place the order. they will be taken to a thank you page with details of their purchase, followed by a confirmation email sent to them.

Improve Registration Forms

May 2018 |

Project Overview

The main goals are:

  • Design a super short form to capture user's email and adjust display rules accordingly to get additional information based on new vs existing user
  • Conduct a research to improve layout and functionality
  • Speed up the registration process and improve conversion rate

My Role

  • Research best practices for form layout and validation
  • Conduct competitive analysis
  • Adjust functionality based on different types of registration if necessary
  • Create mockups showing all steps and communicate functionality with developers

Issues

  • The only way to capture leads were to demand a lot of data from users
  • Lack of instant validation for user error (inline validation)
  • Left align labels of form fields are problematic for international languages with longer text and leads to breaking form layout
  • No visual indication of required vs optional fields

Solution

  • Design a short form with only email field and show the longer form if the user is new
  • Conduct research/competitive analysis and come up with best practices for form functionality
  • Improve UI and functionality based on findings

Challenges

  • Quest forms are tied to several analytics tools and changing their functionality may require many complex updates in the backend
  • Comprehensive regression testing is required once the implementation is complete to make sure data is captured correctly
  • Monitor conversions accurately

Design Process Overview

Research

I started by reading several articles and research papers to find out the primary rules of designing an efficient form. Followings are some the best practices I captured:

  • Explicitly mark both required and optional fields
  • live inline validation has seen an explosive growth in adoption
  • The shorter the form, the better
  • Avoid defaulting fields to certain values unless you believe a large portion of your users will select that value
  • Use input masks where necessary
  • Auto-focus first form field in desktop to create an indication of where to start
  • Match keyboard to input type in mobile
  • Use autocomplete where possible to limit typing
  • Use sentence case for labels as they are easier to read
  • Top-aligned labels have the fastest completion time
  • Maintain inline label's placeholder text on focus. Floating labels is the way to go in this case
  • Avoid generic words for submit button
  • Avoid reset button and use the main CTA in the place that has the most convenient position for clicking in the viewport

Mockup

Based on my findings I have come up with the above solution including the following functionalities: (you can view the old form for reference)

  • Created a short form to capture users email address. If the user is existing, no more information will be required, otherwise, the second form will be presented to capture additional information
  • Used inline floaty labels to solve the issue with international localities and improve the speed of filling out the forms
  • Marked required and optional fields using *
  • Defined visual indicators for interactions (blue for focus, red for errors, green check marks for complete) to make the form more intuitive
  • Introduced inline live validation to
    • make locating errors easier
    • improve the speed of fixing errors while the field context is fresh in users minds
    • avoid skipping a required field unintentionally
  • Used sentence case labels
  • Used centered aligned orange CTA button for submission
