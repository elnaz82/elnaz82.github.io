Quest E-Commerce Website Redesign

May 2018 |

Project Deliverables

  • Identify interaction pain points during customer journey
  • Gather requirements based on data
  • Research e-commerce best practices
  • User flow diagram & wireframes
  • Pixel perfect mockups (Desktop, Tablet, Mobile)
  • Interaction design
  • Deliver assets
  • Developer's handoff using Zeplin

My Role

This project has been done entirely by myself. Some of the main activities are:

  • Understand the current user flow
  • Study actual users feedback/existing data
  • Work closely with e-commerce colleagues and stakeholders to capture edge cases
  • Research e-commerce best practices using Baymard Institute studies
  • Create new user flow based on findings
  • Sketch ideas for the checkout process
  • Arrange critique sessions and iteration on mocks
  • UI testing

Goal (Why?)

Quest e-commerce web app has remained unchanged for a couple of years after it's separation from Dell.

Its redesign will not only help maintain consistent branding across sites but also will create opportunities to improve existing UX flaws which in turn will:

  • Simplify the process of online software shopping
  • Accelerate checkout flow by providing better navigation
  • Support sales by validating and supporting sales team communication
  • Increase company profitability from online sales
  • Helps to organize information hierarchy and contributes to accessible content
  • Efficient interaction will support company philosophy to provide great service and make it easy to do business with.

Audience (Who?)

Quest is a global IT solutions provider to many companies across the world. Here are some specific audiences that interact with Quest e-commerce website:

  • IT professionals. Developers, IT managers, and directors who are responsible to ensure viable service to their customers at all times. (tech savvy)
  • Stakeholders. They are constantly promoting their products and announce new features
  • Partners. This group includes organizations that architect, integrate, implement, deploy and operate the systems that enable Quest’s mutual customers. They also promote Quest products to their audience and receive deals from Quest.
  • Sales reps. Colleagues who assist customers with enterprise sales or any information that is missing online.

In this project, our main focus was IT professionals who purchase our products online.

Our secondary important users were stakeholders to benefit from sale increase and meet their financial goals.

Customer’s context & needs (When & Where?)

Our audience can use the web app from anywhere, office, coffee shop, co-working spaces, airports, etc. They can use their mobile, tablet, or laptop to discover products and most likely make a purchase while they are in front of their laptops/desktops.

They are aware of what their needs are for the business, the problems they are facing and looking for the right solution to meet their budget. They already had a discussion with their technical team and leadership to figure out all the details and money to spend.

They might feel a bit stressed and anxious about the issues affecting their services and want to find the best possible solution as quickly as possible. They might also be excited about using this new solution they are about to buy.

Here are some of the needs they have (user stories):

  • As an IT professional, I need to have access to all the product specs so that I can map them to my service requirements
  • As an international customer, I need to clearly see the prices in my local currency so that I can decide if they meet my budget
  • As a stakeholder, I need to be able to provide enough information to help customers make the right decision and purchase a product so that I can meet my ROI metrics
  • As a customer, I need to be able to check out quickly and easily once I made a decision so that I can start using the product straight away
  • As a customer, I need to receive delivery details and a reference number so that I can refer to in case of any issues

Idea (What?) & Solution

The solution is to redesign existing web app and having it responsive too so that customers can have a consistent experience in all touch points.

Here is a breakdown of user tasks:

  • Go to the e-commerce website
  • Search for a product
  • Find a product
  • View product details
  • Adjust quantity and licensing options
  • Add to cart
  • Proceed to checkout
  • Specify shipping address
  • Provide payment
  • Review order
  • Place order
  • Receive confirmation email along with the status of the order
  • Receive the order
  • Install the product
  • Get support if needed
  • Receive relevant promotions and updates about the product they've purchased

Project/Request Planer

Sep 2017 |

Project Deliverables

This was an enhancement project to Quests existing home-grown CMS hosting the entire website and a ticketing system to manage and plan projects and all the requests associated with each project type to meet deadlines.

  • User flow
  • Iteration on mockups based on constant user feedback
  • Mockups
  • Prototype
  • Testing and validating the concept
  • Interaction design

My Role

This project was a team effort. I worked on the project/request planner feature that was replaced the old style project management tool.

  • Understand the current user flow
  • Work closely with BA to understand the depth of current environment and functionality
  • User interviews to identify main tasks and barriers
  • Competitive analysis to compare and identify missing features or functionalities that can make working with the current product easier and faster
  • Concept testing with users to make sure the new design will satisfy their needs
  • Create mockups and a prototype to show the flow clearly and test with users
  • Several iterations on mocks
  • Interaction design (drag and drop functionality, constant feedback on progress and status of tasks)
  • UI testing
  • Creating a list of common elements and patterns in an uber spreadsheet to build a style guide for designing internal systems

Goal (Why?)

Almost all quest employees need to interact with our internal ticketing system in order to get theier work done.

Interviewing different users in the company revealed that enhancement to the existing system will address the following issues and therefore will improve productivity:

  • Provides a guide to choose the right requests for the project, removing the guesswork, and users no longer need to use Excel checklists.
  • Improve project planning via the Project Request Planner.
  • Improve functional team views of incoming work, avoiding last minute requests and escalations.
  • Display the progression of opening requests and required approvals.
  • Improve data visibility by displaying project phases, SLA, and the calculated end date.
  • Improve project saving capabilities.
  • Establishes a platform to build and update guided projects in the future without IT.

Audience (Who?)

Almost all Quest employees interact with this platform to either request work from other team members or track the pending task list in their bucket.

Following are some of the main groups to name a few:

  • Project managers. Responsible to create a project and all the tasks required from start to finish. They also need to assign tasks to the right teams and keep track of the timeline to meet deadlines and identify risks.
  • Scrum owners. This group is responsible to organize and prioritize tasks
  • Individual task owners. All the team members who are responsible to complete the tasks assigned to them. They also need to prioritize their daily work based on urgency.
  • Directors. They need to work with project managers and monitor reports to ensure deadlines are met and identify any risks or issues in the processes for improvement.

In this project, our main focus was on project managers, agile team members, and individual task owners.

Customer’s context & needs (When & Where?)

All users are interacting with this platform every day or night depending on how much work they need to deliver by a certain date. They may feel stressed or tired from the amount of work they have and want to quickly see all the tasks assigned to them. They want to spend less time on this environment and more time on their actual tasks.

This platform is only available on the desktop.

Here are some of the needs they have (user stories):

  • As a project manager, I need to quickly create the right type of project and get all the necessary tasks automatically created and assigned to proper team members so that I avoid the guesswork and create the wrong tasks.
  • As an Individual task owner, I need to have a visual hierarchy of the tasks assigned to me, filtered by their urgency so that I don't miss deadlines and split my time appropriately among tasks.
  • As a director, I need to monitor the progress of the open projects and identify the risks that may cause delays to meet deadlines, so that I can plan for them in advance.
  • As a project manager, I should be able to see the status of the project and identify roadblocks to avoid missing deadlines.

Idea (What?) & Solution

Talking to users of the platform and listing all the main features that were important to the majority of them, we came up with a new feature to be implemented as an enhancement to improve user experience and efficiency of how we work every day.

Following are some of the tasks that different user groups will perform on this platform:

  • Add a correct type of project/request
  • Monitor progress of different parameters of the project
  • Break projects into different phases
  • Each phase will include required and optional tasks along with SLA time in a card (color coded) view for easier scanning. New tasks can also be added to each phase if needed
  • Requests can be added to each task
  • Additional section to manage requests with more advanced filtering options to quickly identify requests that need urgent attention

Form Research

I started by reading several articles and research papers to find out the primary rules for designing an efficient form. Followings are some of the best practices I captured:

  • Explicitly mark both required and optional fields
  • Live inline validation has seen an explosive growth in adoption
  • The shorter the form, the better
  • Avoid defaulting fields to certain values unless you believe a large portion of your users will select that value
  • Use input masks where necessary
  • Auto-focus first form field in desktop to create an indication of where to start
  • Match keyboard to input type in mobile
  • Use autocomplete where possible to limit typing
  • Use sentence case for labels as they are easier to read
  • Top-aligned labels have the fastest completion time
  • Maintain inline label's placeholder text on focus. (Floating labels is the way to go in this case)
  • Avoid generic words for submit button
  • Avoid reset button and use the main CTA in the place that has the most convenient position for clicking in the viewport

Form User Flow

Based on my findings and working with the business analyst about the requirements of the flow I have come up with the following flow/style for the new form:

  • A short form to capture users email address. If the user is existing, no more information will be required, otherwise, the second form will be presented to capture additional information
  • Inline floaty labels to solve the issue with international localities text labels and improve the speed of filling out the forms
  • Required and optional fields are identifiable through *
  • Visual indicators for interactions (blue for focus, red for errors, green check marks for complete) to make the form more intuitive
  • Inline live validation to:
    • make locating errors easier
    • improve the speed of fixing errors while the field context is fresh in users minds
    • avoid skipping a required field unintentionally
  • Sentence case labels since they are easier to read
  • Centered aligned orange CTA button for larger touch area (mobile friendly)

Design System

Oct 2016 |

Project Deliverables

  • Research a tool to build brand and style guide
  • Identify UI components based on the Atomic design concept
  • Study best practices to create a design system
  • Train different teams to use the new tool to create guidelines for their teams (creative team, developers, copywriters, multimedia team)

My Role

This project was a team effort.

Here are the highlights of my role:

  • Research to find out best practices to build a robust design library
  • Define taxonomy and information architecture using card sorting research method with different teams
  • Work with UI developers to identify UI elements and categorize them to elements (atoms), components (molecules), patterns (organisms), templates, and pages
  • Contribute building Quest web style guide and UI patterns on Frontify
  • Build UI library using Sketch and create symbols for reusable elements for the design team
  • Create guidelines to update & use the UI library

Goal (Why?)

When Quest was part of Dell company all employees had a central place to go and get the information about design system from Dell design library.

After becoming an independent company with a new brand, the need to have this information gathered in one place was even more crucial than before (especially for third-party vendors). This was the main reason for doing this project to:

  • Create a consistent experience and voice across platforms
  • Create visibility to anyone touching the site or other environments of Quest and communicate design standards
  • Create a solid framework that everyone can benefit from

Audience (Who?)

The main audience is basically everyone who needs to touch areas related to Quest such as website, graphics, multimedia, and content. This group includes designers, developers, copywriters, creative team & multimedia folks, project managers, third-party vendors, and product owners.

Users are mainly going to use this environment when working on their tasks. They need to have a single place to make sure complying with Quest branding, voice, and standards to create a distinctive look and feel that is immediately recognizable as Quest.

Customer’s context & needs (When & Where?)

Most of the users are going to refer to this environment while working on different tasks and projects.

It will be effective for them to quickly access the information they need to apply to their day-to-day tasks and don't have to worry about finding them in a file that might not be the latest version.

That way they can spend less time searching and more time working on tasks that matter. Here are some of the needs they may have:

  • Logo standards
  • Brand colors and restrictions to combine them (accessibility standards)
  • Typography, iconography, photography, and media
  • Specifications for different UI elements, functionality, and motion
  • Tone and voice for writing copy
  • Guidelines for marketing communication

Idea (What?) & Solution

The solution was to create a central library as a knowledge base for design and branding using Frontify platform, that way:

  • Teams will have a dedicated area to define standards for their work
  • It will be easy to maintain and edit information using an intuitive portal
  • All information is always available to share and fresh
  • Branding will be consistent on all domains

Research & Analysis

Uber spreadsheet for UI elements

  • Includes all the UI elements categorized as elements, components, and patterns
  • Each element will have a short description of their usage, constraints, and functionalities
  • An example of a page which is using the pattern to help visualize its look and feel

Style guide best practices

Studied brand guidelines in the industry to create a standard taxonomy for the brand style guide and UI pattern library

Building the Platform

Web style guide

Contributed building Quest web style guide and UI pattern library through tangible terminology

UI design library using Sketch

  • Built all the elements from the uber spreadsheet previously defined by design & development teams
  • Created symbols and built Quest library in Sketch so that designers can easily drop common elements to their mockups without spending time rebuilding them. (This would also be a great way to have consistent measures and specifications for mockups across the board)

Measure Success (How?)

Some of the activities that we are going to track are:

  • Active participation & collaboration of team members
  • Efficiency (time spent to do a task by reusing available components)
  • Consistent branding in all aspects

Critique (What could be done better?)

  • Upgrade platform license to give access to more users
  • Have resources to maintain code on this platform so that remote team can easily grab the snippets of code when needed
  • Keep building and evolving the platform as needed so that we create a robust library for everyone to benefit from using it
